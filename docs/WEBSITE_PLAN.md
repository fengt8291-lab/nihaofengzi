# 《你好丰子个人数字主页开发方案 v1.1》

> 历史规划说明：这是旧版历史规划，已被当前以田丰本人为主体的信息架构替代。不得再使用本文件生成导航、路由、Hero 或公开内容；以下历史正文保留不改。

## 一、方案摘要与产品定位

### 1. 产品目标

将 nihaofengzi.top 建设为一个长期成长的个人数字空间，集中展示：

- AI 时代的个人创造力探索
- 人类与 AI 的协作方式
- AI 产品与独立开发实践
- 数字化管理与相关研究
- 持续学习、研究和创造的过程

网站不把“本科生”作为核心身份标签。教育背景保留在 About 和 Resume 中，作为经历与研究方向的真实上下文，不主导首页定位。

访问者应快速形成三个认知：

- “你好丰子”代表一个持续探索 AI 时代个人成长与创造方式的人。
- 这里既有思考和研究，也有真实的产品实践。
- 表达真实、克制、有个人判断力，值得进一步交流或合作。

### 2. 核心受众

- 第一优先：项目伙伴、导师、招聘方、潜在合作者。
- 第二优先：关注 AI 产品、人机协作、个人生产力和数字化管理的探索者。
- 第三优先：AI Soul、DailyTasks、今夜补救等具体项目的潜在用户。

首页优先回答“我在探索什么、如何实践、正在做什么、为什么值得进一步了解”，而不是先展示学历和技能列表。

### 3. 与普通简历网站的区别

普通简历网站以经历罗列为核心；你好丰子以“探索—研究—创造—实践”为主线：

- Resume 只是可信度证明之一。
- Projects 展示如何发现问题、形成判断并把想法落地。
- Research 展示对人机协作和数字化管理问题的系统理解。
- Now 展示当下真实关注的问题，避免网站成为静态档案。
- 所有项目允许处于实验、原型或进行中状态，不包装虚假成果。

## 二、“你好丰子”品牌概念

### 1. 品牌含义

“你好丰子”来源于名字中的“丰”，表达：

- 丰盛：不断拓展经验和视野
- 积累：长期沉淀项目、研究与思考
- 成长：接受不完整，持续迭代
- 连接：连接商业、AI、产品与个人实践

品牌语气应当真诚、年轻、理性、克制，避免企业宣传口吻或虚构专家身份。

### 2. “丰”字视觉策略

“丰”字不直接设计成具象 Logo，也不在页面中反复作为巨大汉字装饰，而是成为抽象视觉系统：

- 横向结构代表学习、研究、创造等不同探索层。
- 纵向结构代表个人成长主线以及不同领域之间的连接。
- 通过细线、节点、留白、分栏和卡片对齐关系隐性呈现。
- 可扩展为网格、成长轨迹、内容关系图和页面转场语言。
- 即使移除文字“丰”，视觉系统仍应成立，避免符号使用过度。

品牌文字标识使用“你好丰子 / NIHAO FENGZI”，保持简洁排版，不额外创建复杂图形商标。

## 三、网站信息架构

### 1. 全局导航

桌面端采用轻量吸顶导航：

- 你好丰子
- About
- Projects
- Research
- Resume
- 联系入口

移动端使用简洁折叠菜单。Blog 在至少具备 3 篇完整文章前不进入主导航。

### 2. 页面结构

| 页面 | 路径 | 页面作用 |
|---|---|---|
| 首页 | `/` | 建立品牌定位，展示当前探索、精选项目、研究方向、个人形象与联系入口 |
| About | `/about` | 讲述个人经历、关注方向、思考形成过程和当前状态 |
| Projects | `/projects` | 展示 AI Soul、DailyTasks、今夜补救及未来产品实践 |
| 项目详情 | `/projects/[slug]` | 按问题、判断、过程、方案、状态与反思呈现案例 |
| Research | `/research` | 聚合人机协作、AI 员工、一人公司、数字化管理和 ESG 研究 |
| 研究详情 | `/research/[slug]` | 展示研究问题、方法、阶段成果、限制和关联材料 |
| Resume | `/resume` | 提供移动端网页简历和 PDF 下载 |
| Blog（未来） | `/blog`、`/blog/[slug]` | 记录 AI 实践、产品思考和学习过程 |
| 404 | `/404` | 提供符合品牌语气的错误提示和返回路径 |

### 3. 内容边界

- About：解释“我为什么关注这些问题”。
- Projects：证明“我如何把思考转化为产品实践”。
- Research：体现“我如何系统理解问题”。
- Resume：快速查阅教育、经历、能力和联系方式。
- Now：展示当前阶段，不承担完整履历功能。

## 四、首页设计方案

### 1. Hero 首屏

Hero 以个人品牌和核心命题为中心，不突出学生身份。

建议内容：

**品牌提示**

> NIHAO FENGZI / 你好丰子

**核心标题**

> 探索 AI 时代的个人创造力。

**辅助说明**

> 围绕人机协作与 AI 产品实践，我持续把思考转化为项目、研究和真实体验。

**品牌行动线**

> 学习 · 研究 · 创造 · 实践

**行动入口**

- 主按钮：查看我的项目
- 次按钮：了解我
- 文字入口：Currently Exploring

Hero 不使用技能标签墙、求职状态或“某某专业本科生”作为视觉重点。

### 2. Hero 视觉表达

- 使用“丰”字结构衍生的抽象成长坐标。
- 三层横向轨迹代表创造力、人机协作、产品实践。
- 一条纵向轨迹连接项目、研究和个人成长。
- 通过光点、细线、透明度和缓慢呼吸形成轻微动态。
- 鼠标响应幅度保持克制；移动端使用静态或低动态版本。
- 不使用 AI 大脑、机器人、发光球体等通用科技素材。

### 3. 首页内容顺序

1. **Hero**：品牌名称、核心命题、行动入口。
2. **Now / Currently Exploring**：展示当前真实关注。
3. **Selected Projects**：按照指定顺序展示三个核心项目。
4. **Research & Thinking**：展示人机协作、AI 员工和一人公司研究。
5. **About Snapshot**：个人照片、背景摘要与成长路径。
6. **Contact CTA**：邀请交流、合作或讨论。
7. **Footer**：联系方式、社交链接、版权与备案信息。

不使用强制整屏滚动和滚动劫持。每个区块集中表达一个主题，通过留白、字体尺度和明暗变化形成节奏。

### 4. Now / Currently Exploring

Now 模块作为首页长期保留内容，建议展示 3 个可定期更新的方向：

- AI Agent 如何进入个人工作流
- 人类与 AI 如何形成更自然的协作关系
- 一个人如何利用 AI 完成产品探索与创造

模块包含“最后更新于”信息，后续通过一份结构化数据更新，不需要修改页面组件。

### 5. Projects 展示顺序

#### 第一：AI Soul

作为最能代表 AI 探索方向的首要项目，使用最大视觉面积。

重点表达：

- 对 AI 工具和人机交互方式的实验
- AI 如何成为个人思考和创造的协作者
- 项目当前阶段与真实探索过程

#### 第二：DailyTasks

作为连接数字世界与现实生活的产品实践。

重点表达：

- 信息过载和现实低连接问题
- 轻量任务与积极反馈机制
- 从问题洞察到产品方案的过程

#### 第三：今夜补救

作为具体生活场景中的行为辅助产品。

重点表达：

- 作息失控后的恢复需求
- 睡眠管理与行为调整
- 个性化建议的产品思路

项目卡片必须显示项目状态、个人角色、问题摘要和详情入口，不只显示技术栈。

## 五、个人照片展示策略

### 1. 目标

照片用于增强真实感、亲近感和个人辨识度，不用于制造商业领袖或成熟企业家的形象。

整体感觉：

- 专业，但不过度正式
- 年轻，但不随意
- 真实，但经过良好构图
- 有思考感，而不是标准证件照

### 2. 首页使用方式

Hero 仍以文字和抽象品牌视觉为主，不放置占据半屏的大型人物照片。

个人照片放在首页 About Snapshot：

- 使用自然环境或简洁室内背景。
- 采用半身或环境人像，而非纯证件照。
- 可搭配一句简短的个人陈述。
- 图片与文字采用非对称留白构图。
- 移动端保持人物主体完整，避免自动裁切面部。

### 3. About 页面使用方式

About 首段可使用一张更完整的个人照片，并在后续内容中少量加入学习、研究或产品实践场景照片。

优先级：

1. 清晰自然的个人肖像
2. 真实工作、学习或讨论场景
3. 项目过程和实体记录
4. 纯装饰图片

### 4. 图片标准

- 优先使用自然光、干净背景和简单服装。
- 避免夸张西装照、商业棚拍效果、重度磨皮和 AI 生成人像。
- 原图建议至少 2000px 长边，并保留横版、竖版和方形裁切空间。
- 所有照片由本人确认公开范围。
- 图片提供合理替代文本、固定尺寸和现代格式。
- 没有合适照片时先使用品牌视觉占位，不使用低质量临时照片上线。

## 六、技术架构

### 1. 总体策略

- Astro 负责页面、路由、内容渲染与静态生成。
- TypeScript 使用严格模式。
- Tailwind CSS 管理设计令牌、布局和响应式样式。
- React 只用于需要客户端交互的局部 Islands。
- Framer Motion 首期仅用于 Hero 抽象视觉和少量重点动画。
- 普通卡片 hover、导航和基础显隐优先使用 CSS。
- v1 不引入数据库、CMS、SSR、登录系统或全站状态管理。

Astro 静态构建默认输出 `dist/`，适合当前 Nginx 部署；内容集合用于在构建阶段加载和校验 Markdown 数据。[Astro 部署文档](https://docs.astro.build/en/guides/deploy/)、[Content Collections](https://docs.astro.build/en/guides/content-collections/)

### 2. 建议结构

```text
src/
├── assets/                 # 由 Astro 优化的图片和项目素材
├── components/
│   ├── layout/             # Header、Footer、SEO
│   ├── ui/                 # Button、Container、Section、Tag
│   ├── home/               # Hero、Now、SelectedProjects
│   ├── projects/           # ProjectCard、ProjectMeta
│   ├── research/           # ResearchCard、TopicList
│   └── motion/             # React / Framer Motion Islands
├── content/
│   ├── projects/
│   ├── research/
│   └── blog/
├── data/                   # 站点配置、Now、导航、简历
├── layouts/
├── pages/
└── styles/
public/
├── media/
├── resume/
├── favicon/
└── robots.txt
```

```ts
  cover

```

首页通过 `order` 固定排序：

1. AI Soul
2. DailyTasks
3. 今夜补救

```ts

  title
  slug
  summary

  topics[]

  publicationUrl?
  attachment?
```

```ts

  title

```

```ts

  title
  slug
  description

  cover?
  draft

```

项目、研究和未来博客正文使用 Markdown。首页、列表页和详情页从同一内容源生成，避免重复维护。

### 4. 组件边界

- `BaseLayout`：全局骨架和页面元数据。
- `SEOHead`：标题、描述、canonical、社交分享与结构化数据。
- `Hero`：品牌文案和行动入口。
- `HeroVisual`：独立 React Island；无 JavaScript时保留静态版本。
- `CurrentlyExploring`：从结构化数据渲染。
- `ProjectCard`：统一项目信息和状态表达。
- `PortraitSection`：处理个人照片、响应式裁切与替代文本。
- `ResearchCard`：统一研究主题与阶段信息。

### 5. 图片和视频

- 普通内容图片放入 `src/assets`，使用 Astro 图片能力处理。
- PDF、favicon、原样视频和 poster 放入 `public`。
- v1 不使用首屏背景视频。
- 视频必须具备 poster、暂停控制、静音策略和移动端降级。
- 图片固定宽高，避免布局跳动。
- 资源明显增长后再评估对象存储或 CDN。

### 6. SEO

- 统一标题模板、描述、canonical 和社交分享图。
- 首页使用适用的 `Person` 结构化数据。
- 项目详情使用适用的 `CreativeWork` 或 `SoftwareApplication` 信息。
- 生成 sitemap、robots.txt 和 404 页面。
- 草稿页面不进入 sitemap。
- 品牌统一为“你好丰子 / Nihao Fengzi”，实施时修正 README 中现有的“你好疯子”笔误。

## 七、视觉与交互设计

### 1. 色彩

- 页面背景：`#F5F5F7`
- 卡片背景：`#FFFFFF`
- 主文字：`#1D1D1F`
- 次文字：`#6E6E73`
- 主强调色：`#0066CC`
- 成长辅助色：`#789174`
- 深色区：`#0A0A0B`
- 深色区文字：`#F5F5F7`

辅助色只用于 Now、研究标签和轨迹节点，不进行大面积堆叠。

### 2. 字体

优先使用系统字体：

```text
-apple-system, BlinkMacSystemFont, "SF Pro Display",
"PingFang SC", "Microsoft YaHei", sans-serif
```

Hero 使用大字号和有限行宽；正文保持舒适行高，不使用过细字重。

### 3. 版式

- 内容最大宽度约 1200–1280px。
- 长正文宽度约 680–760px。
- 使用 8px 间距体系。
- 卡片圆角控制在 16–28px。
- 主要依靠留白、排版和背景层次，减少明显阴影。
- “丰”字横纵关系贯穿网格、分割线和内容对齐。

### 4. 动画

- 页面进入：透明度与 12–24px 位移，300–500ms。
- 卡片反馈：160–240ms，最大位移约 4px。
- Hero 轨迹：缓慢、低幅度，不影响文字阅读。
- 全面支持 `prefers-reduced-motion`。
- 不使用滚动劫持、自定义鼠标、无意义计数器和夸张视差。

## 八、Phase 1 部署与更新流程

### 1. 首期流程

```text
本地开发
→ 本地检查与构建
→ Git 提交
→ 推送 GitHub
→ 登录阿里云 Ubuntu
→ 拉取最新代码
→ 安装锁定依赖并构建
→ 更新 /var/www/nihaofengzi/dist
→ 验证网站
```

Phase 1 暂不创建 GitHub Actions 自动部署，不配置自动连接生产服务器。

### 2. 手动部署要求

- `main` 分支作为生产基准。
- 部署前确认工作区已提交，GitHub 与本地提交一致。
- 服务器通过 `git pull` 获取代码。
- 使用锁文件对应的确定性安装命令安装依赖。
- 构建成功后才替换生产 `dist`。
- 更新前备份上一版本的 `dist` 或记录上一生产提交。
- 部署失败时保留原站点，回滚到上一提交重新构建。
- 每次部署记录提交哈希、日期和结果。

### 3. Nginx 与域名

- `https://nihaofengzi.top` 作为 canonical。
- `www` 永久重定向到主域名。
- Nginx root 指向 `/var/www/nihaofengzi/dist`。
- 使用 `try_files $uri $uri/ =404` 服务 Astro 多页面站点。
- HTML 使用短缓存，带哈希的资源使用长期缓存。
- 启用 HTTPS、HTTP/2、压缩、证书自动续期和基础安全响应头。

如果服务器位于中国内地，上线前完成 ICP 备案，并在网站开通后按要求办理公安联网备案和展示备案信息。[阿里云备案说明](https://help.aliyun.com/zh/document_detail/3043304.html)

### 4. 自动部署的后续位置

GitHub Actions 自动检查、自动发布和原子版本切换延后至 Phase 4，待手动流程稳定且服务器权限边界明确后再实施。

## 九、开发路线

### Phase 1：基础网站

- 初始化 Astro、TypeScript、Tailwind、React 和静态构建。
- 建立设计令牌、组件边界和内容集合。
- 完成首页、About、Projects、项目详情、Research、研究详情、Resume、404。
- 首页项目顺序固定为 AI Soul、DailyTasks、今夜补救。
- 完成 Now 模块、个人照片区域、网页简历和 PDF 下载。
- 完成移动端、SEO、sitemap、Nginx 和手动部署流程。

完成标准：主路径完整、内容真实、移动端可用、生产域名稳定访问。

### Phase 2：视觉优化

- 完善“丰”字抽象视觉系统。
- 优化 Hero 品牌表达、项目封面和照片呈现。
- 制作统一的截图、封面和研究视觉规范。
- 加入克制的显隐、hover 和滚动反馈。
- 完成真实设备视觉测试和性能回归。

### Phase 3：高级交互

- 增强 Hero 轨迹的轻量交互。
- 加入统一页面切换和项目案例叙事动画。
- 根据内容需要增加时间线、研究关系图或产品演示。
- 所有高级效果提供移动端和 reduced-motion 降级。
- AI 助手不默认进入该阶段，需单独评估内容、隐私和运行成本。

### Phase 4：持续扩展

- Blog 达到内容门槛后开放导航、RSS、标签和归档。
- 持续增加项目案例、研究记录和阶段复盘。
- 根据维护需求评估轻量 CMS、对象存储和隐私友好分析。
- 手动部署稳定后，再实现 GitHub Actions 自动检查和发布。
- 每季度检查内容、外链、简历、依赖、SEO 和性能。

## 十、测试与验收

### 内容与功能
